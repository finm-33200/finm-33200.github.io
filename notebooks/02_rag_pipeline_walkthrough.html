RAG Pipeline Walkthrough#

Notebook 01 described the data; notebook 03 reports the benchmark numbers. This middle notebook explains how the pipeline itself is wired together – function by function, with small concrete inputs and outputs – so that when the results show up in notebook 03 you already know exactly what was done to produce each number.

Everything here is imported from src/rag_benchmark.py and src/embeddings_index.py. The runnable scripts call OpenAI; this notebook stays mostly offline and uses saved CSV outputs from a previous run to show what the model actually returned. That way the walkthrough is cheap to re-execute and does not require an API key (it does briefly call the embeddings API in Step 3b to embed one query, but that costs a fraction of a cent and can be skipped).

Two flavors of retrieval#

“RAG” just means retrieve text from a corpus, then generate. It does not specify how retrieval works. We benchmark two retrieval families in this homework:

  • Structured retrieval (find_10k_filing): the question already carries a CIK and a fiscal-year-end, so we look up the matching 10-K directly and run a keyword-based chunk extractor over it. This is what the rag_cleaned and rag_raw conditions use.

  • Dense semantic retrieval (semantic_search): the corpus is pre-chunked and every chunk is embedded into a 1536-dim vector with the OpenAI embeddings API. At query time we embed the question and ask FAISS for the top-k nearest chunks across the whole corpus – no CIK/year prefilter. This is what the rag_semantic_cleaned condition uses, and it is what most people mean when they say “classical RAG.”

Both end up calling the same downstream code (build_rag_prompt, run_prompt, extract_number, is_correct). Only the retrieval step changes.

The pipeline at a glance#

For each (company, fiscal year, metric) question we do the same thing in four conditions (baseline, RAG-cleaned, RAG-raw, RAG-semantic-cleaned):

    build_test_cases()              # generate questions from Compustat
            |
            v
    ---- pick a retrieval path (skipped in baseline) ----
      structured:  find_10k_filing()  +  extract_financial_context_*()
      semantic:    semantic_search()  over a prebuilt FAISS index
            |
            v
    build_rag_prompt()              # assemble the prompt
            |
            v
    run_prompt()                    # OpenAI chat.completions call
            |
            v
    extract_number()                # parse a number out of the reply
            |
            v
    is_correct()                    # compare to Compustat ground truth

Step 3: chunk selection#

  1. Keyword + number lines (e.g. “Total revenue $383,285”) and grabs a few lines of surrounding context.

  2. Financial narrative paragraphs that mention at least two financial keywords and contain a dollar figure.

Step 3b: dense semantic retrieval (the “classical RAG” path)#

The structured-lookup retrieval above works because the question already names the company and the fiscal year, and we have a clean index keyed on those. Most production RAG stacks do not have that luxury – the question is free text and there is no obvious join key. So they fall back to semantic search: embed the corpus once, embed the query, and ask a vector index for the most similar chunks.

Three pieces are involved:

  1. A chunker that splits each filing into overlapping passages. chunk_text(text, chunk_size, overlap) is a sliding character window – nothing fancy, just chunk_size characters at a time with overlap characters of overlap so a sentence isn’t sliced in half.

  2. An embedding model that turns each chunk (and later, each query) into a fixed-length vector. We use OpenAI text-embedding-3-small – 1536 dimensions, same API client we already use for the chat completions.

  3. A vector index that, given a query vector, returns the top-k most similar chunk vectors. We use FAISS (IndexFlatIP with L2-normalized vectors, which gives exact cosine similarity over the ~55k chunks in the cleaned corpus).

The build (task_build_embeddings in dodo.py -> python src/embeddings_index.py) chunks all 48 cleaned filings, embeds every chunk in batches, and writes:

  • _data/embeddings/cleaned_chunks.parquet – one row per chunk with metadata (cik, accession, fdate, char_start, char_end, text).

  • _data/embeddings/cleaned_index.faiss – the FAISS index, with row order aligned to the parquet.

We do this once and cache, so subsequent benchmark runs only pay the query-side embedding cost.

A semantic-search query#

semantic_search(client, question, chunks_df, faiss_index, top_k, max_words) embeds the question, calls FAISS for the top-k chunks, and concatenates them up to a word budget. It returns the same shape of dict as find_10k_filing plus a top_k_filings field that lets us inspect which filings the chunks came from.

We need an OpenAI client only to embed the question (the corpus embeddings are already cached). If you do not have an API key handy, skip the next two cells and read the prose – the rest of the notebook does not depend on this call.

Step 5: the LLM call#

run_prompt(client, model_id, prompt, system_msg=...) is a thin wrapper around openai.chat.completions.create. It measures wall time, captures token counts, and returns a dict:

{
    "output":        "...",   # the model's reply text
    "latency_s":     0.87,
    "input_tokens":  1234,
    "output_tokens": 12,
}

We do not call the API here – that happens in src/rag_benchmark.py at build time and the results are cached to CSV. Let’s load the cached results and look at one real reply for the sample question.

df_baseline = pd.read_csv(OUTPUT_DIR / "baseline_results.csv")

Step 6: parse a number out of the reply#

The LLM answers in prose (“Apple’s fiscal 2024 revenue was approximately $383 billion.”) but the grader needs a numeric value in millions of USD. extract_number handles the common shapes:

examples = [
    "Apple's fiscal 2024 revenue was approximately $383 billion.",
    "Net income was 96,995 million.",
    "The total assets figure is $352,755.",
    "Approximately -2,000 million (net loss).",
    "I don't know.",
]
pd.DataFrame({
    "llm_reply": examples,
    "extracted_millions": [extract_number(x) for x in examples],
})
llm_reply extracted_millions
0 Apple's fiscal 2024 revenue was approximately ... 383000.0
1 Net income was 96,995 million. 96995.0
2 The total assets figure is $352,755. 352755.0
3 Approximately -2,000 million (net loss). 2000.0
4 I don't know. NaN

The parser’s preference order is:

  1. "X billion" -> X * 1000 (convert to millions).

  2. "X million" -> X.

  3. Otherwise, fall back to the last numeric token in the string (typically the figure the model quoted last).

It’s deliberately forgiving – these are small models and the prose varies. The 5% and 10% tolerance bands in the next step absorb any rounding.

Step 7: grade the answer#

Where to go from here#

Notebook 03 scales this up: it runs the full grid of questions through all four conditions and reports aggregate accuracy, latency, token usage, and cost. Before opening it, you should now be able to predict the qualitative shape of the results:

  • Baseline is cheap but inaccurate (the model is guessing).

  • RAG-cleaned (structured) is accurate and still cheap (small model, compact context, clean text, exact filing match).

  • RAG-raw (structured) is accurate-ish but costs more per query and sometimes trips on markup – WRDS’s cleaning pipeline earns its keep.

  • RAG-semantic-cleaned trades the exact filing match for a global FAISS lookup. Whether this helps or hurts depends on whether the top-k chunks land on the right company; top_chunk_cik_match in the per-question CSV tells you.

Levers you can pull:

  • Change FINANCIAL_KEYWORDS / MAX_WORDS_* in src/rag_benchmark.py to reshape the keyword chunker.

  • Change CHUNK_SIZE / CHUNK_OVERLAP / EMBED_MODEL in src/embeddings_index.py (then rerun task_build_embeddings) to reshape the semantic chunker.

  • Change top_k / max_words in run_rag_semantic to give the LLM more or fewer retrieved chunks per question.
